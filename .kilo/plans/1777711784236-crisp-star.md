# 🚀 PLAN FINAL OPTIMISÉ (VERSION MASTER / INDUSTRIE) - Predictive Sales Analysis Project

## 🧠 0. POSITIONNEMENT DU PROJET
Ce projet ne se contente pas de créer un modèle de prédiction simple. Il développe un **système Big Data de prédiction de ventes de bout en bout (end-to-end pipeline)**, démontrant les principes du Big Data (Volume, Vitesse, Variété) à travers une architecture multicouche professionnelle.

## Project Overview
This project aims to develop a predictive sales analysis system for a supermarket using historical data. The system will forecast future sales based on customer behavior, seasonal patterns, and promotional activities. The focus is on demonstrating Big Data principles through a layered architecture utilizing Apache Spark for processing and Firebase for serving results.

## Objectives
- Predict future sales using historical data (customers, seasons, promotions)
- Demonstrate Big Data architecture with ingestion, storage, processing, feature store, ML, and serving layers
- Implement end-to-end ML pipeline with Spark MLlib and proper temporal splitting
- Deploy predictions to Firebase for real-time access
- Maximize academic value through proper documentation, advanced feature engineering, and justification of choices
- Position as Time Series Forecasting problem with industrial vision

## Data Sources
### Primary Dataset
- **Kaggle Supermarket Sales Dataset**: Use the "Supermarket Sales" dataset by Alex Huitron (https://www.kaggle.com/datasets/alexhuitron/supermarket-sales) which contains sales transactions with features like:
  - Invoice ID, Branch, City, Customer type, Gender, Product line, Unit price, Quantity, Tax, Total, Date, Time, Payment, COGS, Gross margin percentage, Gross income, Rating

### Additional Data Sources (for Big Data Variety)
- **Holidays Data**: JSON file containing public holidays and special events
- **Weather Data**: API integration (e.g., OpenWeatherMap API) to include weather conditions that might affect sales
- **Promotional Data**: Additional CSV/JSON for ongoing promotions

## Architecture Design

```
[SOURCES]
   ↓
[INGESTION]
   ↓
[DATA LAKE - RAW]
   ↓
[PROCESSING - SPARK ETL]
   ↓
[FEATURE STORE]
   ↓
[ML PIPELINE]
   ↓
[PREDICTIONS]
   ↓
[SERVING - FIREBASE]
   ↓
[VISUALISATION]
```

### 1. Ingestion Layer
- **Purpose**: Collect data from multiple sources via structured pipeline
- **Implementation**:
  - Batch ingestion for static files (CSV, JSON)
  - API ingestion for dynamic data (weather)
- **Tools**: Python scripts with requests library for API, file operations for local data
- **Upgrade**: Data ingestion pipeline concept

### 2. Storage Layer (Data Lake)
- **Purpose**: Store raw data without transformation, with versioning
- **Implementation**: Local directory structure mimicking HDFS/S3
  - `/data-lake/raw/supermarket_sales.csv`
  - `/data-lake/raw/holidays.json`
  - `/data-lake/raw/weather/`
  - `/data-lake/cleaned/`
  - `/data-lake/features/`
- **Justification**: Simulates enterprise Data Lake, maintains data lineage and separation of levels

### 3. Processing Layer (ETL Pipeline)
- **Purpose**: Clean, transform, and prepare data for ML
- **Implementation**: Apache Spark (PySpark) ETL Pipeline with separation:
  - Cleaning: handle missing values, outliers, duplicates
  - Transformation: data integration (join sales with holidays/weather)
  - Feature Engineering: one-hot encoding, date extraction, advanced features (lags, rolling means)
- **Justification**: Handles large datasets efficiently, in-memory processing, structured ETL

### 4. Feature Store
- **Purpose**: Store ML-ready features for reusability and reproducibility
- **Implementation**: Parquet files in `/data-lake/features/sales_features.parquet`
  - Contains features temporelles, encoded variables, interaction features
- **Justification**: Academic gain énorme - réutilisabilité + reproductibilité, industrial standard

### 5. ML/Analysis Layer
- **Purpose**: Train predictive models using proper time series approach
- **Implementation**: Spark MLlib with Pipeline
  - Temporal split (train=past, test=future)
  - Regression algorithms (Linear Regression, Random Forest Regression, theoretical ARIMA/Prophet)
  - Model evaluation and validation with RMSE, MAE, MAPE, R²
  - Hyperparameter tuning
- **Problem Classification**: Time Series Forecasting
- **Pipeline**: Spark ML Pipeline with stages (StringIndexer, OneHotEncoder, VectorAssembler, Model)
- **Justification**: Scientifically correct temporal split, advanced FE, scalable for Big Data

### 6. Serving Layer
- **Purpose**: Store and serve predictions with decoupling
- **Implementation**: Firebase Firestore/Realtime Database
  - Store prediction results in structured NoSQL documents
  - Optional: Simple web dashboard to visualize predictions
- **Data Model**:
  ```json
  {
    "date": "2026-06-01",
    "predicted_sales": 12000,
    "branch": "A",
    "confidence": 0.87
  }
  ```
- **Architecture**: Spark → batch predictions → Firebase; Frontend → Firebase → Dashboard
- **Justification**: NoSQL database for fast access, separates analytical from operational data, explains découplage Processing/Serving
